Detailed Description

Container object holding configuration of an instance of LbmCt.

The caller creates an config instance and sets desired options, and then passes it to LbmCt#start.

Note that the CT object retains a reference to the passed-in configuration object, so it is not recommended to modify options after the LbmCt is created, except for testing purposes. Also, if creating multiple CT objects, it is generally recommended to create separate configuration objects, one for each CT object (unless they all share exactly the same configuration).

◆ setMaxTries()

void com.latencybusters.lbmct.LbmCtConfig.setMaxTries ( int  maxTries)

Override the default value for the configuration object's number of tries.

When a connected source and receiver are connecting or disconnecting, they exchange handshake messages to synchronize state. If a CT source or receiver sends a handshake and expects a response but does not get one within retryIvl milliseconds, the operation will be retried. After maxTries attempts, CT will give up, log an error, and stop retrying.

Parameters
maxTriesNumber of times to attempt handshakes.

◆ setPreDelivery()

void com.latencybusters.lbmct.LbmCtConfig.setPreDelivery ( int  preDelivery)

Override the default value for the configuration object's received message pre-connected delivery behavior.

Normally, a Connected Topics Receiver will not deliver messages received prior to a connection being established. Setting preDelivery to 1 enables the CT receiver to deliver messages received outside of the connected state. This can enable a CT receiver to get messages from a non-CT source, although this use case is not recommended. See .

Parameters
preDelivery1=deliver messages outside of connection; 0=do not deliver.

◆ CT_CONFIG_DEFAULT_MAX_TRIES

final int com.latencybusters.lbmct.LbmCtConfig.CT_CONFIG_DEFAULT_MAX_TRIES = 35
static

Initial value for maxTries; try a handshake operation this many times before giving up.

Be aware that if a network outage lasts longer than maxTries * retryIvl, but the transport session does not time out, the connection could fail to be established. The defaults were chosen to be longer than the default transport session timeouts for LBT-RM and LBT-RU. See setMaxTries.
